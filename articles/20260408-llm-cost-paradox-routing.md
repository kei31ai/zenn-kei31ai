## LLMの推論コストは下がった。3年で約300倍

2023年3月、GPT-4（OpenAI製）がリリースされたとき、入力トークン単価は**約$30〜$36/百万トークン**だった（DeployBase, Andrew Ng氏のX投稿より。入力トークン価格）。

それが2024年にはGPT-4oで約$4/百万トークン。2026年現在、GPT-4相当の性能を持つモデルが$0.40以下で使える。3年間でざっくり**約300倍**のコスト低下だ（TokenCost AI Price Index調べ）。

**なぜここまで下がったのか？** ガートナーは2026年3月25日のプレスリリースで以下の要因を挙げている。

- 半導体・インフラ効率の向上
- モデル設計の革新（蒸留・量子化等）
- チップ稼働率の向上
- 推論特化シリコンの普及
- エッジデバイスへの展開

同レポートでは「2030年までに1兆パラメータLLMの推論コストが2025年比で90%超削減」「2022年比で最大100倍のコスト効率化」を予測している。

![トークン単価の推移（2023年〜2026年、2030年予測）](/images/20260408-llm-cost-paradox-routing/token-price-trend.jpg)

これだけ見ると「AIは安くなる一方では？」と思うかもしれない。しかし現実は逆だ。

## それでもAI請求書は増える。なぜか

**LLM推論コストとは、AIモデルに問い合わせるたびに発生するAPI費用のことで、トークン単価×消費トークン数で決まる。**

単価が300倍下がったのに総コストが増えるのは、**消費量の増加が単価低下を上回っているから**だ。

ガートナーの同レポートによると、エージェントモデルは標準的なチャットボットと比べて**1タスクあたり5〜30倍**のトークンを消費する。

なぜそんなに増えるのか。エージェントは「1回聞いて終わり」ではない。タスク分解、ツール呼び出し、中間確認、エラー処理、最終回答の生成——これらを繰り返す。1つのタスクで10〜20回のLLMコールが発生するケースも珍しくない（oplexa.com調べ）。

**具体例1: Claude Code**

AnthropicのClaude Code（Anthropic製AIコーディングツール）公式ドキュメントによると、マルチエージェントチームを使うとプランモードで通常セッション比**約7倍**のトークンを消費する。開発者1人あたり1日平均$6、90%のユーザーは$12/日以下——月換算で$100〜200になる。

単価が安くなっても、エージェント化で消費量が7倍になれば総コストは上がる。シンプルな構造だ。

```
総コスト = トークン単価 × 消費トークン数

例）単価が 1/10 になっても、消費量が 7〜30倍になれば:
    総コスト = 0.1 × 7〜30 = 0.7〜3倍
```

![単価↓ × 消費量↑↑ = 総コスト↑の構造図](/images/20260408-llm-cost-paradox-routing/cost-paradox-structure.jpg)

ガートナーもこの点を明示している。「トークン単価の下落よりも消費量の増加が速く、全体的な推論コストは増加する見通しであり、企業顧客にはコスト削減が全額還元されない」と。

## ルーティングが解答になる理由

コスト増加への答えとしてガートナーが推奨するのが**モデルルーティング**だ。

> 定型・高頻度タスクは効率的な小型/特化モデルへ割り当て、フロンティアモデルは高マージン・複雑推論タスクに限定する
> （Gartner, 2026-03-25プレスリリース）

つまり「全部GPT-4に投げる」のをやめて、タスクの複雑さに応じてモデルを使い分ける設計だ。

**具体例2: RouteLLM**

UC Berkeley・Anyscale・Canvaの研究者が開発した**RouteLLM**（ICLR 2025掲載）は、この考え方を実装したフレームワークだ。

MT Benchの結果では、GPT-4（OpenAI製）の**95%の性能を維持しながら、コストを最大85%削減**できることが示された。実装はOpenAI互換のAPIラッパーとして動作し、既存のコードに最小限の変更で導入できる。

```python
from routellm.controller import Controller

client = Controller(
    routers=["mf"],
    strong_model="gpt-4-1106-preview",
    weak_model="anyscale/mistralai/Mistral-7B-Instruct-v0.1",
)
# 以降はOpenAI APIと同じインターフェースで使用可能
response = client.chat.completions.create(model="router", messages=[...])
```

またガートナーの2025年4月9日付のプレスリリースでは「2027年までに企業は汎用LLMの**3倍の頻度**でタスク特化型AIモデルを使うようになる」と予測している。Small language modelの使い分けが標準になっていく流れだ。

**実践フレームワーク:**

| タスクタイプ | 特性 | 推奨モデル |
|---|---|---|
| 定型・反復 | FAQ応答、フォーマット変換、分類 | SLM・タスク特化モデル |
| 中程度 | 要約、翻訳、コード補完 | 中規模モデル |
| 複雑推論 | 設計判断、創作、マルチステップ推論 | フロンティアモデル |

ルーティング設計の順序:
1. **タスク棚卸し**: エージェントが何回LLMを呼ぶかをログで計測する
2. **タスク分類**: 定型/中程度/複雑推論の3カテゴリに分ける
3. **モデル割り当て**: 各カテゴリにコストと品質を考慮してモデルを選ぶ
4. **ルーティング実装**: RouteLLM等のフレームワークを使うか、自前でスコアリングロジックを書く
5. **計測と改善**: 総コスト（単価×消費量）をタスクごとに記録し、サイクルを回す

重要なのは**エージェント化する前の設計段階で決める**こと。ワークフローを組み終えた後からルーティングを後付けするのは難しい。

## Q&A

**Q: すべてのタスクをSLMに任せればいいのでは？**

A: 定型・反復作業はSLMで十分だが、複雑な推論・創造的なアウトプット・意思決定が絡むタスクはフロンティアモデルが必要。RouteLLMが示したのは「95%の品質を維持できる」ことであって「100%置き換えられる」ではない。ルーティングとは、フロンティアモデルを使う**場面を絞る**設計だ。

**Q: ルーティング設計はいつから始めるべきか？**

A: エージェント化する前の設計段階から。どのタスクをどのモデルに任せるかは、アーキテクチャ設計と同じレイヤーで決める。LLMコスト最適化は後付けより先付けの方が圧倒的に効く。

## まとめ

- **LLM推論コスト**は2030年までに90%超削減される（Gartner予測）
- しかし**AIエージェントコスト**の総額は増加する。エージェント化でトークン消費量が5〜30倍になるから
- 競争軸は「安いトークン」から「**賢いモデルルーティング設計**」へ移行している
- RouteLLMのように、GPT-4の95%性能を維持しながら85%コスト削減する手法は実在する
- 設計はエージェント化の前から。後付けは難しい

「トークンが安くなった」で安心するのではなく、どのタスクにどのモデルを使うかの設計が、今後のAI活用コストを左右する。
