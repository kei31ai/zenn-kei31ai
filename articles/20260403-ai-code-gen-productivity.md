「GitHub CopilotでPR数が増えた」「コーディングが速くなった」という声が増えている。数字を見ると確かにそうだ。でも、チームの開発速度が上がった実感が薄い人も多いはずだ。

この「感覚のズレ」は気のせいじゃない。

2024年から2026年初頭にかけて公開された4つの大規模研究が、同じ現象を異なる角度から指摘している。個人レベルでは改善が見られる。だが組織全体の生産性との間に、構造的なギャップがある。

## 4つの研究が示す数字

| 研究 | 規模 | 個人レベル | 組織レベル |
|-----|-----|-----------|-----------|
| GitHub × Accenture RCT | 企業内開発者（RCT: ランダム化比較試験） | PR数 **+8.69%**、マージ率 **+15%** | 測定なし |
| METR（経験者OS開発研究） | 開発者16名、246タスク | 速度 **−19%**（経験者） | 測定なし |
| Faros AI（10,000人超） | 開発者10,000人超、1,255チーム | タスク完了 **+21%** | 企業レベルの改善と**相関なし** |
| DX社（400社縦断） | 400社、2024/11〜2026/2 | PR **+9.97%** | コーディング以外は変わらず |

![4つの研究が示す個人vs組織の数字](/images/20260403-ai-code-gen-productivity/fig1-research-comparison.jpg)

一見バラバラに見えるこの4つのデータに、共通した構造がある。

## なぜ1: PRレビューがボトルネックになる

Faros AIが開発者10,000人超のデータを分析した結果、AI採用チームではタスク完了が21%増え、マージされたPRが98%増えた。個人の出力は確実に上がっている。

ところが、同時にPRレビュー時間が**91%増加**し、PRの平均サイズが**154%拡大**した。

コードを書く速度を上げると、人間が行うレビューという工程に負荷が集中する。Amdahl's Law（アムダールの法則）とは「どんなに部分を最適化しても、システム全体の速度は最も遅い部分に制約される」という原則だ。これをソフトウェア開発に当てはめると、AIがコーディングを速くしたぶん、レビューというボトルネックが以前より目立つようになる。

これがFaros AIの報告した「AIプロダクティビティパラドックス」の正体だ。

![なぜ個人の速度向上が組織に伝わらないのか](/images/20260403-ai-code-gen-productivity/fig2-bottleneck-flow.jpg)

> 参照: [The AI Productivity Paradox Research Report - Faros AI](https://www.faros.ai/blog/ai-software-engineering)

## なぜ2: AIはコーディングしか速くしない

DX社が400社を対象に1年以上追跡した調査では、AI利用量が65%増加したにもかかわらず、PRスループットの改善は**約10%**にとどまった。

研究者の言葉が核心をついている。

> 「4日のタスクが3日になるかもしれないが、それはPRを3倍出すことを意味しない」

コーディングは開発プロセス全体の一部に過ぎない。要件定義、設計、レビュー、テスト、デプロイ、チーム間の調整——これらは依然として人間が担う。AIがコーディング速度を上げても、これらの工程にはほぼ影響しない。

> 参照: [AI productivity gains are 10%, not 10x - DX](https://getdx.com/blog/ai-productivity-gains-are-10-percent-not-10x/)

## なぜ3: 経験者ほど逆効果になることがある

最も反直感的なデータがMETRの研究だ。

経験豊富なオープンソース開発者16名を対象にしたRCT（ランダム化比較試験）で、AI使用を許可した場合のタスク完了時間を測定した。結果は**19%の速度低下**。驚くべきことに、参加者自身は「20%速くなった」と感じていた。

> 開発者が予測した効果: +24%
> 実際の結果: −19%

METRはこの乖離の要因として5つを挙げているが、大規模で成熟したコードベース（平均22,000スター以上、100万行以上）では、AIが生成するコードが既存の文脈から外れることが多いという点が大きい。

経験者はコードのコンテキストを深く理解しているため、AI生成コードの修正・検証コストが予想より高くつく。

> 参照: [Measuring the Impact of Early-2025 AI on Experienced Open-Source Developer Productivity - METR](https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/)

## 組織改善なしでAI導入しても意味がない

DORA 2025レポートは、AIを「既存のエンジニアリング条件の乗数」と表現している。高性能なチームはさらに強くなり、プロセスが断片化した組織の弱さはより露呈する。

GitHub × Accentureの研究でも、PR数やマージ率は改善したが、組織全体の生産性を直接測定したデータは含まれていない。8.69%のPR増加が、ビジネス成果にどう繋がったかは示されていない。

> 参照: [Research: Quantifying GitHub Copilot's impact in the enterprise with Accenture - GitHub](https://github.blog/news-insights/research/research-quantifying-github-copilots-impact-in-the-enterprise-with-accenture/)

個人の速度を上げることで、組織の問題が表面化するだけになるケースがある。

:::details Q&A
**Q: GitHub CopilotやCursorを使えば生産性は上がりますか？**

A: 個人の特定タスク（単純なコーディング、テスト作成、ドキュメント生成など）では改善が期待できます。ただし、組織全体の生産性はPRレビュープロセスや計画・調整の仕組みを整備しないと向上しにくいことが複数の研究で示されています。「ツールを入れれば速くなる」ではなく、「ツールが速くした部分の周辺工程が追いつくか」が問われます。

**Q: METRの−19%は全員に当てはまりますか？**

A: 当てはまりません。METRの研究は「経験豊富な開発者」「大規模OSSリポジトリ」「特定のタスク」という条件下での結果です。単純なタスクや新しいコードベースでは別の結果になる可能性があります。研究チーム自身も適用範囲の限界を明記しています。
:::

## まとめ

4つの研究から見えてくる構造は一貫している。

- AIはコーディング速度を改善する（+8〜21%程度）
- ただし経験者では逆効果になるケースもある（−19%）
- 個人の改善が組織全体の生産性改善に直結しない
- ボトルネックはコーディングではなく、レビュー・調整・計画にある

AIコーディングツールを入れる前に問うべきことがある。「PRレビューの仕組みは整っているか」「コーディング以外の工程に余裕はあるか」——道具より仕組みを先に整えることが、組織全体の生産性に繋がる近道だ。
