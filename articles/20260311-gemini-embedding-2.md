Gemini Embedding 2とは、Googleが2026年3月10日にリリースした初の完全マルチモーダル埋め込みモデルです。テキスト・画像・動画・音声・PDFを1つのベクトル空間に埋め込めます。ただし、旧モデルとembedding spaceが非互換で、既存のベクトルインデックスは全て再構築が必要です。この記事では、Gemini Embedding 2の技術仕様と、再埋め込み（re-embedding）が意味する実務的な影響を解説します。

## Gemini Embedding 2とは

Gemini Embedding 2（モデル名: `gemini-embedding-2-preview`）とは、Google DeepMindが開発したマルチモーダル埋め込みモデルのことです。「埋め込み（embedding）」とは、テキストや画像などのデータを数値のベクトルに変換する技術で、ベクトル同士の距離で「意味の近さ」を計算し、検索やレコメンドに使います。

Gemini Embedding 2は、以下の入力モダリティに対応しています。

- **テキスト**: 最大8,192トークン、100以上の言語に対応
- **画像**: 最大6枚（PNG, JPEG）
- **動画**: 最大128秒（MP4, MOV）
- **音声**: 最大80秒（MP3, WAV）
- **PDF**: 最大6ページ

出力のベクトル次元は128から3,072まで柔軟に選べます。Matryoshka Representation Learning（MRL）という手法を使っていて、用途に応じて次元数を調整できます。推奨は768、1,536、3,072の3段階です（[Gemini API docs](https://ai.google.dev/gemini-api/docs/embeddings)）。

## なぜマルチモーダル埋め込みが重要なのか

従来のベクトル検索では、モダリティごとにパイプラインが分かれていました。

- テキスト検索 → テキスト用embeddingモデル
- 画像検索 → 画像用embeddingモデル（または画像をキャプション化してテキストで検索）
- 音声検索 → 音声を文字起こし → テキストで検索
- PDF検索 → OCR → テキストで検索

これだと、「商品画像に似た動画シーンを探す」「音声の内容に近いPDFを探す」といったクロスモーダル検索が構造的にできませんでした。モダリティをまたぐたびにテキストへの変換が挟まり、情報が落ちていたからです。

Gemini Embedding 2は、全モダリティを1つのベクトル空間に写像します。テキスト・画像・動画・音声・PDFのどれを入力しても、同じ空間上のベクトルが返ってくるため、異なるモダリティ間の検索が直接できるようになります。

RAG（Retrieval-Augmented Generation）を構築している開発者にとって、これは検索パイプラインの設計が根本的に変わる話です。モダリティごとのインデックスを統合できれば、検索ロジックがシンプルになり、メンテナンスコストも下がります（[Google Blog](https://blog.google/innovation-and-ai/models-and-research/gemini-models/gemini-embedding-2/)）。

## 再埋め込み問題 -- ベクトル検索基盤の作り直し

Gemini Embedding 2で見落とせないのが、Googleのドキュメントに明記されているこの一文です。

> gemini-embedding-001 と gemini-embedding-2-preview の embedding space は互換性がない。移行する場合、ドキュメントもクエリも全て再埋め込みが必要

これは「新しいモデルが出たから差し替えよう」で済む話ではありません。embedding spaceが変わるということは、既存のベクトルインデックスが丸ごと使えなくなります。

具体的に何が起きるかを整理します。

- **インデックス再構築**: 100万件のドキュメントがあれば、100万件全てをGemini Embedding 2で再計算する必要がある
- **コスト**: API呼び出し回数 x 料金。バッチ処理を使えばデフォルト料金の50%で済むが、大規模データではそれでも数万円単位になる
- **ダウンタイム**: 再埋め込み処理中は新旧インデックスの整合性を管理する必要がある
- **評価セットの再設計**: 検索品質の評価基準も新しいembedding spaceに合わせて作り直す必要がある

新規でRAGシステムを構築するなら、最初からGemini Embedding 2を選べばいいだけです。でも、本番環境で数十万件以上のドキュメントをすでに埋め込んでいるチームにとっては、切り替えの判断はコスト見積もりとセットになります（[Gemini API changelog](https://ai.google.dev/gemini-api/docs/changelog)）。

## Q&A

**Q: 既存のgemini-embedding-001から乗り換えるべきですか？**

A: ケースによります。新規プロジェクトならGemini Embedding 2を選ぶのがいいです。マルチモーダル対応、MRLによる次元調整、100言語以上のサポートなど、機能面では大幅に強化されています。一方、すでに大規模なインデックスを運用中の場合は、再埋め込みのコスト（API料金 + 処理時間 + 評価セット再設計）を見積もってから判断するのがおすすめです。バッチ処理（50%割引）を使えばAPI料金は抑えられます。

## まとめ

- **Gemini Embedding 2**は、テキスト・画像・動画・音声・PDFを1つのベクトル空間に統合するGemini初のマルチモーダル埋め込みモデルです
- マルチモーダル対応の本質は「パイプラインの統合」です。モダリティごとにバラバラだった検索基盤を1つのAPIにまとめることで、クロスモーダル検索が直接できるようになります
- 旧モデルとembedding spaceが非互換のため、移行には再埋め込みが必須です。本番環境では、コスト見積もりと評価セットの再設計を含めた移行計画が必要です

生成AIのモデル更新は目立ちますが、あとから効いてくるのは検索基盤の変更です。RAGの品質を本気で上げたいなら、モデル選びより先にベクトル検索の設計を見直すタイミングかもしれません。
